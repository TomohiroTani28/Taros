### 6.5.3 パウリフレーム追跡の実装

パウリフレーム追跡（Pauli Frame Tracking, PFT）は、v3.0の制御アーキテクチャの根幹である。v2.0では各fusionゲートの結果に基づくリアルタイムフィードフォワードが必要とされていたが、Critical Reviewにおいて2μsサイクル内でのフィードフォワード完了が非現実的であると指摘された。

**PFTの原理**:

クリフォードゲート（パウリ・アダマール・CNOT・S）はパウリ群の共役作用で完全に記述できるため、測定結果に依存するパウリ修正は物理的に適用する必要がなく、古典的なソフトウェア上でパウリフレーム（各キュービットに付随するパウリ演算子の記録）を更新するだけでよい [Knill 2005, Nature 434, 39; Raussendorf+ 2003]。

**PFT実装**:

| 処理段階 | 処理内容 | レイテンシ | 実装 |
|---|---|---|---|
| 1. 検出信号受信 | SNSPD信号のデジタイズ | < 5 ns | 比較器 + TDC |
| 2. 衝突判定 | 行列読み出しのデコード | < 20 ns | FPGA (専用ロジック) |
| 3. Fusion結果判定 | 2検出器の結果からfusion成功/失敗を判定 | < 10 ns | FPGA |
| 4. パウリフレーム更新 | クリフォードテーブル参照→フレーム更新 | < 50 ns | FPGA (BRAM lookup) |
| 5. 次サイクル準備 | EO制御信号の生成（必要な場合のみ） | < 100 ns | FPGA → DAC → EO |
| **合計（データ面）** | | **< 200 ns** | |
| 6. 復号（蒸留面のみ） | MWPM / Union-Find | < 1 ms | GPU |
| 7. フィードフォワード（蒸留面のみ） | マジック状態の受理/棄却判定 | HCF遅延（2μs）内 | FPGA |

**リアルタイムフィードフォワードが必要な箇所**: マジック状態蒸留のみ。蒸留プロトコルでは、シンドローム測定結果に基づいてマジック状態を受理または棄却する判定が必要であり、この判定結果を次のfusion操作に反映しなければならない。この遅延をHCF 600m（2μs）で吸収する設計となっている（9.2節で詳述）。

**Rack 7-8 消費電力**: ~5 kW（FPGA: ~0.6kW, GPU: ~2.8kW, ネットワーク・ストレージ: ~1.6kW）
**Rack 7-8 設置面積**: 標準19インチラック 2本（42U × 2）
